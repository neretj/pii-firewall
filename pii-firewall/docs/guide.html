Privacy Firewall

A domain-aware PII anonymization library for AI applications. Detects, redacts, and restores personal data around LLM calls — in 55+ languages.

Domain-aware anonymization
Auto language detection
Reversible pseudonymization
GDPR right to be forgotten
Multiple detection backends

📦 Installation

The package is named pii-firewall and the Python module is privacy_firewall. You import from privacy_firewall.

Option A — Minimal (pattern-based only, no ML dependencies)

# No extra system dependencies required
pip install pii-firewall

⚠️ Minimal install uses regex-only detection. Sufficient for well-structured data (IDs, emails, phones) but misses NER entities like people or organization names.

Option B — Recommended (Presidio + language detection)

pip install "pii-firewall[presidio,langdetect]"

# Then download the spaCy language model(s) you need:
python -m spacy download es_core_news_sm   # Spanish
python -m spacy download en_core_web_sm   # English
python -m spacy download fr_core_news_sm  # French

Option C — Full features

pip install "pii-firewall[all]"

Option D — Local development (editable install from this repo)

cd artifacts/pii-firewall
pip install -e ".[presidio,langdetect]"

Tip: If you only need one language and want to skip auto-detection, install without [langdetect] and always pass language="es" when creating a firewall. This avoids the langdetect dependency entirely.

⚡ Quick Start

The absolute minimum to anonymize a text:

from privacy_firewall import create_firewall

# Create a firewall for your domain
firewall = create_firewall("healthcare")

# Required context: identifies the tenant, case, and conversation thread
context = {
    "tenant_id": "hospital-001",
    "case_id":   "patient-123",
    "thread_id": "consultation-1",
    "actor_id":  "doctor-456",
}

result = firewall.process(
    text="Ana García, 43 años, hipertensión. Prescripción: enalapril 10mg.",
    context=context,
)

print(result.sanitized_text)
# → "PERSON_1, [AGE_40-49], hipertensión. enalapril 10mg."
#    Medical terms are KEPT. Names and ages are anonymized.

context is how threads are tracked. Using the same thread_id across multiple messages allows the vault to maintain consistent token mappings — PERSON_1 always refers to the same real person within that thread.

Why these context fields exist

FieldWhat it doesWhy it matters
tenant_id Organization boundary Hard isolation across customers. Same token in different tenants never shares mapping.
thread_id Conversation bucket Main isolation key for mappings. Same thread keeps stable tokens across turns.
case_id Grouping key for related threads Useful for GDPR and operations: lets you group multiple threads for one patient/case in your app layer.
actor_id Audit actor identifier Compliance/audit metadata. Does not change token mapping logic.

Practical rule: thread_id is the key that actually drives mapping continuity. case_id helps you organize related threads (for retention/GDPR workflows), and actor_id is for traceability.

Without auto language detection

# If you avoid langdetect, pass language explicitly
firewall = create_firewall("healthcare", language="es")

🔄 How It Works

Privacy Firewall acts as an anonymization layer that sits between your application and the LLM. Text is cleaned before going to the model, and restored after the model responds.

1. Input Raw text
"Ana García, DNI 12345678A"
→
2. Detect Find PII
Presidio / regex / etc.
→
3. Anonymize Replace tokens
"PERSON_1, NATIONAL_ID_1"
→
4. LLM Model sees
anonymized text only
→
5. Rehydrate Restore names
"Ana García, DNI..."

The vault stores the mapping between tokens and real values per thread. If you call process() (which calls the LLM), rehydration is automatic. If you only need anonymization (e.g., you're in a proxy microservice), use anonymize() and rehydrate() separately.

🏥 Domain Profiles

The domain profile controls what counts as sensitive and what action to take on each entity type. Some entities should be kept (they're useful to the LLM), others must be redacted.

🏥 healthcare

  • Keeps: diagnoses, medications, procedures
  • Redacts: names, IDs, addresses
  • Generalizes: ages (43 → 40-49), dates

💼 finance

  • Keeps: amounts, account types, credit scores
  • Masks: credit cards (4111...1111)
  • Pseudonymizes: account numbers (reversible)

⚖️ legal

  • Keeps: case numbers, statutes
  • Pseudonymizes: party names
  • Generalizes: all dates to year only

🌐 generic

  • Sensible defaults for any domain
  • Good starting point for custom profiles
firewall = create_firewall("healthcare")  # or "finance", "legal", "generic"

Custom Profiles

from privacy_firewall import create_custom_profile, EntityDisposition, DispositionAction

profile = create_custom_profile("my_domain")

profile.add_disposition(EntityDisposition(
    entity_type="EMPLOYEE_ID",
    action=DispositionAction.REDACT,
    confidence_threshold=0.9,
))

profile.add_disposition(EntityDisposition(
    entity_type="CASE_NUMBER",
    action=DispositionAction.KEEP,    # pass through as-is
    confidence_threshold=0.8,
))

firewall = create_firewall("generic", profile=profile)

🔍 Detection Backends

Backend Install extra Strengths Latency
regex base (no extra) IDs, emails, phones, credit cards. Zero dependencies. < 1 ms
presidio [presidio] + spaCy model Named entities (persons, orgs, locations). Best balance. 50–200 ms
transformers [transformers] Domain NER (BioBERT, FinBERT). Highest accuracy. 100–500 ms
gliner [gliner] Zero-shot NER, no fine-tuning needed. 100–400 ms
hybrid [presidio,langdetect] Combines regex + presidio for maximum coverage. 50–250 ms 
# Recommended: Presidio for best speed/accuracy balance
firewall = create_firewall("healthcare", detector_backend="presidio")

# Zero dependency option
firewall = create_firewall("healthcare", detector_backend="regex")

🎚️ Disposition Actions

Each entity type in a profile has a disposition — what to do with it when found.

ActionResultReversible?
KEEP Pass through unchanged (e.g. medical diagnosis) N/A
PSEUDONYMIZE Ana García → PERSON_1 (stored in vault) ✅ Yes
REDACT john@example.com → [REDACTED] ❌ No
GENERALIZE 43 años → [AGE_40-49] ❌ No
MASK 4111111111111111 → 4111...1111 ❌ No
HASH SHA-256 hash of the value ❌ No
SUPPRESS Remove the entity entirely (no replacement) ❌ No

Why only PSEUDONYMIZE is reversible: it stores a token-to-original mapping in the vault (for example, PERSON_1 → Ana Garcia). The other actions intentionally destroy or alter the original value without storing it for recovery: REDACT replaces with a constant marker, GENERALIZE keeps only a range/category, MASK keeps partial characters, HASH is one-way, and SUPPRESS removes the value entirely. KEEP is marked N/A because nothing is transformed, so there is nothing to "rehydrate".

If you need recoverability for a field (email, phone, account, person name), choose PSEUDONYMIZE for that entity type. If you choose any irreversible action, recovery is not possible by design.

🗄️ Vault & Rehydration

The vault stores the mapping from token (PERSON_1) to original value (Ana García) per tenant + case + thread. This enables rehydration after the LLM responds.

In-Memory Vault (default)

Data lives in memory, wiped when the process restarts. Fine for development and short-lived microservices.

SQLite Vault (persistent)

from privacy_firewall import create_firewall, SQLiteMappingVault

vault = SQLiteMappingVault("privacy_vault.db")
firewall = create_firewall("healthcare", vault=vault)

Data survives restarts. Store privacy_vault.db on a persistent volume (e.g. Fly.io volume).

Manual rehydration

# 1. Anonymize (before LLM call)
anon_result = firewall.anonymize(text=input_text, context=context)
clean_text  = anon_result.sanitized_text

# 2. Call your LLM (any provider)
llm_response = my_llm.generate(clean_text)

# 3. Rehydrate (after LLM response)
final_text = firewall.rehydrate(text=llm_response, context=context)
print(final_text)  # Original names restored

🧩 SDK Pattern (provider-agnostic)

PrivacyFirewallSDK wraps the firewall with convenience methods ideal for building LLM wrappers or middleware. 

from privacy_firewall import PrivacyFirewallSDK

sdk = PrivacyFirewallSDK.create(
    domain="healthcare",
    detector_backend="presidio",
    language="es",           # omit for auto-detect
)

context = {
    "tenant_id": "hospital-001",
    "case_id":   "patient-123",
    "thread_id": "thread-1",
    "actor_id":  "doctor-456",
}

# ── Anonymize only ──────────────────────────────────────────────
result = sdk.anonymize_text(text="Contact Ana García at ana@clinic.es", context=context)
print(result.sanitized_text)  # "Contact PERSON_1 at EMAIL_1"

# ── Rehydrate after LLM ──────────────────────────────────────────
llm_out = "Please confirm with PERSON_1 at EMAIL_1"
final   = sdk.rehydrate_text(text=llm_out, context=context)
print(final)  # "Please confirm with Ana García at ana@clinic.es"

# ── Full round-trip (anonymize → LLM → rehydrate) ─────────────
def my_llm(prompt: str) -> str:
    # Your LLM client here
    return f"Analysed: {prompt}"

result = sdk.secure_call(
    text="Ana García, 43 años, hipertensión",
    context=context,
    llm_client=my_llm,
)
print(result.sanitized_text)  # What the LLM saw
print(result.final_text)      # Restored output shown to user

Anonymize/rehydrate whole JSON payloads

payload = {
    "patient": "Ana García",
    "notes": ["DNI: 12345678A", "Email: ana@clinic.es"]
}

safe_payload = sdk.anonymize_payload(payload=payload, context=context)
# deep-walks strings recursively — lists and nested dicts work too

🔌 Microservice Pattern

The best production pattern is a dedicated Python NLP microservice that exposes HTTP endpoints. Your main backend calls it — language or framework agnostic.

Minimal FastAPI microservice

# main.py — Privacy Firewall Microservice
import os, subprocess, sys, logging
from contextlib import asynccontextmanager
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field

try:
    from privacy_firewall import PrivacyFirewallSDK
except Exception:
    PrivacyFirewallSDK = None

logger = logging.getLogger("nlp")
privacy_sdk = None

def _ensure_spacy_model(model: str):
    import spacy
    try:
        spacy.load(model); return
    except: pass
    subprocess.run([sys.executable, "-m", "spacy", "download", model], check=True)

def _bootstrap():
    global privacy_sdk
    if PrivacyFirewallSDK is None: return
    models = os.getenv("PII_FIREWALL_SPACY_MODELS", "es_core_news_sm").split(",")
    for m in models: _ensure_spacy_model(m.strip())
    privacy_sdk = PrivacyFirewallSDK.create(
        domain=os.getenv("PII_FIREWALL_DOMAIN", "healthcare"),
        language=os.getenv("PII_FIREWALL_LANGUAGE", "es") or None,
        detector_backend=os.getenv("PII_FIREWALL_DETECTOR_BACKEND", "presidio"),
    )

@asynccontextmanager
async def lifespan(app):
    _bootstrap()
    yield

app = FastAPI(lifespan=lifespan)

class SanitizeReq(BaseModel):
    text: str
    context: dict

@app.post("/privacy/sanitize")
async def sanitize(req: SanitizeReq):
    if not privacy_sdk: raise HTTPException(503, "Privacy runtime not available")
    result = privacy_sdk.anonymize_text(text=req.text, context=req.context)
    return {"sanitized_text": result.sanitized_text}

@app.post("/privacy/rehydrate")
async def rehydrate(req: SanitizeReq):
    if not privacy_sdk: raise HTTPException(503, "Privacy runtime not available")
    final = privacy_sdk.rehydrate_text(text=req.text, context=req.context)
    return {"final_text": final}

@app.get("/health")
async def health(): return {"status": "ok", "privacy_loaded": privacy_sdk is not None}

Calling from Node.js / TypeScript backend

// nlp.service.ts
const NLP_URL = process.env.NLP_SERVICE_URL ?? "http://localhost:8000";

export async function sanitizeText(text: string, context: Record<string, string>) {
  const res = await fetch(`${NLP_URL}/privacy/sanitize`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ text, context }),
  });
  const { sanitized_text } = await res.json();
  return sanitized_text as string;
}

🔧 Custom PII Detectors

You can plug in your own entity detectors at runtime — no config files, just code. There are two approaches depending on whether you need a simple regex rule or a full NLP/ML model.

Option A (regex patterns) works with any detection backend. Option B (custom recognizers) requires detector_backend="presidio".

Option A — Regex / rule-based (any backend)

The quickest way. No extra dependencies — just pass a regex string.

# One-liner helper
firewall.add_custom_regex(
    entity_type="EMPLOYEE_ID",
    regex=r"\bEMP-\d{6}\b",
    locales=["GLOBAL"],           # apply to all languages
    confidence=0.95,
    context_words=["employee", "staff"],
    disposition_action="redact",   # keep / redact / pseudonymize …
)

# Or build the full EntityPattern object for more control
import re
from privacy_firewall.patterns.catalog import EntityPattern

firewall.add_custom_pattern(EntityPattern(
    entity_type="CASE_NUMBER",
    locale="ES",                    # or "GLOBAL" for all languages
    pattern=re.compile(r"\bEXP-\d{4}/\d{6}\b"),
    confidence=0.98,
    context_words=("expediente", "exp"),
    description="Spanish legal case number",
))

Option B — Custom NLP / ML model (Presidio backend)

Pass a Presidio EntityRecognizer instance when creating the firewall. This is the right approach for spaCy models, transformers, or any custom heuristic that needs access to full token sequences. 

from privacy_firewall import create_firewall
from privacy_firewall.presidio_integration import create_custom_recognizer

# Helper: wraps a list of regexes into a Presidio PatternRecognizer
employee_recognizer = create_custom_recognizer(
    entity_type="EMPLOYEE_ID",
    patterns=[r"\bEMP\d{6}\b"],
    context_words=["employee", "badge"],
    score=0.9,
)

firewall = create_firewall(
    domain="generic",
    detector_backend="presidio",   # required for custom recognizers
    custom_recognizers=[employee_recognizer],
)

For a fully custom ML-based recognizer, subclass Presidio's EntityRecognizer directly:

from presidio_analyzer import EntityRecognizer, RecognizerResult

class MyModelRecognizer(EntityRecognizer):
    """Wraps any ML model as a Presidio recognizer."""

    def load(self): ...

    def analyze(self, text, entities, nlp_artifacts):
        results = []
        for span in my_model.predict(text):   # your model call here
            results.append(RecognizerResult(
                entity_type="CUSTOM_ENTITY",
                start=span.start,
                end=span.end,
                score=span.confidence,
            ))
        return results

firewall = create_firewall(
    domain="generic",
    detector_backend="presidio",
    custom_recognizers=[MyModelRecognizer(supported_entities=["CUSTOM_ENTITY"])],
)

Which option to choose?

ScenarioApproach
Regex or rule-based custom entityOption A — add_custom_regex / add_custom_pattern
Locale-specific ID format (new country)Option A with the matching locale code
Existing HuggingFace / spaCy NER modelOption B — wrap in EntityRecognizer subclass
Complex heuristic or external API callOption B — implement analyze() freely

Combine with regex patterns (Presidio hybrid)

If you need the HF model to run alongside regex patterns in the same pipeline, wrap it as a Presidio recognizer and use detector_backend="presidio":

from presidio_analyzer import EntityRecognizer, RecognizerResult
from transformers import pipeline

class HFPIIRecognizer(EntityRecognizer):
    def __init__(self, model_id: str):
        super().__init__(supported_entities=["PERSON", "ORGANIZATION", "LOCATION"])
        self._pipe = pipeline("ner", model=model_id, aggregation_strategy="simple")

    def load(self): ...

    def analyze(self, text, entities, nlp_artifacts):
        return [
            RecognizerResult(
                entity_type=span["entity_group"],
                start=span["start"],
                end=span["end"],
                score=span["score"],
            )
            for span in self._pipe(text)
        ]

firewall = create_firewall(
    "healthcare",
    detector_backend="presidio",
    custom_recognizers=[HFPIIRecognizer("dslim/bert-base-NER")],
)

Tip: Use detector_backend="transformers" for quick benchmarking of a single model. Switch to the Presidio wrapper when you need to combine it with locale-specific regex patterns for maximum coverage.

🔐 GDPR — Right to Be Forgotten

Remove all stored mappings for a specific case or thread. This satisfies Art. 17 GDPR right to erasure. 

# Forget all token↔value mappings for this thread
deleted = firewall.forget(
    tenant_id="hospital-001",
    case_id="patient-123",
    thread_id="consultation-1",
)
print(f"Deleted {deleted} mappings")
# After this, rehydration will not restore original values for this thread.

Note: Only the vault mappings are deleted. Any already-sent LLM responses or logs your application stored are outside the library's scope.
